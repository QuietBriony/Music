# drum-floor由来運用ルール（実装ではなく運用 docs）

drum-floorから取り込む運用知見を、Music統合段階の運用規約として固定します。  
実装ファイルには直接反映せず、運用手順とPRレビュー時の確認項目として扱います。

## 1. Limiter優先

- 最終段にLimiterを置く前提を維持する
- 目標:
  - 音割れ/クリッピングの一次回避
  - 突発的なピークに対しても破裂を起こしにくい構造
- 事前チェック:
  - メインアウト直前に制限ノードが存在すること
  - `-1 dB` 前後基準値を崩さない

## 2. CPU上昇時の縮退（degrade）

### 基本方針
- CPUが上昇したら、音質より安定を優先
- `Reverb / Delay / Distortion` を段階的にオフ or 低負荷化
- ループ生成の密度を下げる

### 実施優先順（候補）
1. リバーブウェット量を落とす
2. ディレイ/フィードバックを落とす
3. 一部の生成トリガーを間引く
4. それでも改善しない場合はUI側で重い機能をOFF

### 回復条件
- 5〜10秒でCPUが収束したら正常値へ段階復帰
- 回復が難しければ低負荷状態を維持

## 3. ライブ／操作中の事故回避

### 絶対ルール
- 「鳴らなくなった時」は、まず Audio Module 再接続/再初期化を優先
- 同時に「ページ再読込」に進む（最終手段）
- 音割れ時は即Limiter再確認と、過負荷エフェクト停止

### 本番チェック（短縮版）
1. 音源I/O固定
2. サンプルレート確認（例: 48k）
3. バッファサイズ（中位優先）
4. Limiter ON
5. 10分連続再生試験

### リカバリ（30秒運用）
- 無音/クリップを検知したら
  1. デバイス再選択
  2. 再読込

## 4. PR運用（Music統合向け）

- 1PRあたり1目的（例: `updateEnergy` 結合のみ）
- 例外ログと再現条件をREADME/PR本文に記載
- 事故が起きる操作（例: 高負荷同時開始、端末再帰起動）を明示

## 5. 反映方針

- 実装仕様としてコード化しない
- Music側で運用ノートとして参照し、必要な時のみ実装に落とす
- docsの更新が必要な場合は別PRとして扱い、統合PRとは分離
